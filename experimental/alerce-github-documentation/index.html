Organization Overview

ALeRCE is a Chilean-led astronomical alert broker designed to characterize and classify alert streams from large etendue surveys such as ZTF and LSST. The GitHub organization hosts repositories spanning the full stack: from real-time Kafka-based pipeline steps to web frontends, Python clients, ML classifiers, and science use-case notebooks.

Pipeline (Monorepo & Legacy Steps)

The core real-time alert processing pipeline. Since 2023, all pipeline steps are consolidated into the pipeline monorepo. Older standalone step repos are now archived. The pipeline ingests alerts via Apache Kafka, assigns ALeRCE IDs, computes photometric corrections, extracts features, and runs ML classifiers. It supports multi-survey processing (ZTF, ATLAS, and future LSST).

Pipeline Step Flow (Detail)

When a new alert arrives from a survey (e.g., ZTF), it passes through the following steps in order within the pipeline monorepo. Each step is a Kafka consumer/producer that can be independently scaled:

1. Sorting Hat

Assigns an ALeRCE ID (aid) to incoming alerts by performing spatial crossmatch against known objects. If the alert position matches an existing object (within a configurable radius), it is associated with that object's aid. Otherwise, a new aid is created. This step is survey-agnostic and supports multi-survey object identification.

2. Previous Candidates

Decodes the extra_fields payload (ZTF-specific) to extract the previous candidates embedded in the alert packet. Produces a structured output containing the aid, a list of detections, and a list of non-detections.

3. Stamp Classifier

Applies a CNN-based classifier to the science, reference, and difference image stamps from the first detection. Classifies into a 5-class taxonomy: AGN, SN, Variable Star, Asteroid, Bogus. A 2025 version (stamp_classifier_2025_step) is included in the monorepo with updated models.

4. Lightcurve Step

Retrieves the complete lightcurve of the associated object from the database, merging new detections/non-detections with existing ones, deduplicating, and resolving conflicts.

5. Correction

Applies photometric corrections to transform difference magnitudes into apparent magnitudes. A multistream variant (correction_multistream_ztf_step) handles ZTF-specific corrections in the multi-survey context.

6. Magnitude Statistics

Computes per-band statistics over the corrected lightcurve: mean magnitude, amplitude, dmdt, number of detections, and other descriptors used downstream by classifiers and the web interfaces.

7. Crossmatch (Xmatch)

Performs positional crossmatch against external catalogs, primarily ALLWISE, to obtain infrared photometry. Also queries Gaia, SDSS, and PanSTARRS for additional multi-wavelength information.

8. Feature Extraction

Computes a rich set of features from the lightcurve using lc_classifier's feature extractors: periodograms (P4J, mhps), autoregressive models, parametric fits, and statistical features. These require a minimum number of detections per band (typically 6).

9. Light Curve Classifier

Applies the hierarchical Random Forest classifier to the extracted features. Produces class probabilities across a detailed taxonomy including: Cepheids, RR Lyrae, Eclipsing Binaries, LPVs, Delta Scuti, SNe, AGN, QSOs, YSOs, Blazars, CVs, and others. Only classifies objects with sufficient feature data.

Auxiliary Steps

S3 Archiving: Backs up raw AVRO files to AWS S3. Scribe: Handles asynchronous database writes via CQRS pattern. Watchlist: Checks incoming alerts against user-defined watchlists and sends notifications. Reflector: Re-emits alerts to Kafka output topics. Early Classification and LC Anomaly: Additional classification steps for early-stage and anomaly detection.

Multi-Survey Architecture

ALeRCE is designed to handle multiple astronomical alert streams simultaneously. The following repositories and components are specifically relevant to multi-survey processing:

Multi-Survey Repositories

The pipeline monorepo contains survey-agnostic steps (Sorting Hat, Features, LC Classifier) alongside survey-specific steps (e.g., correction_multistream_ztf_step). The alerce_client provides both a ZTF-specific API and a multi-survey API. The web-services backend supports querying across surveys. The atlas_realtime repo handles ATLAS stream ingestion, and the stamp_service stores stamps from multiple surveys.

ZTF-Only vs. Multi-Survey

Historically, all ALeRCE repositories were ZTF-focused. The transition to multi-survey processing (in preparation for LSST) involved refactoring the Sorting Hat to assign cross-survey aid identifiers, creating survey-agnostic database schemas, and updating the client and web interfaces to support survey filtering. The ATAT transformer model is designed from the ground up for multi-survey classification.

For ALeRCE Users (Astronomers)

If you are an astronomer wanting to use ALeRCE data, these are the repositories you should focus on:

Getting Started

usecases — Start here. Contains Jupyter notebooks for variable stars, supernovae, AGN, and general exploration. Includes workshop materials and Colab-compatible tutorials. Clone with git clone --depth=1 for faster download.

alerce_client — Install the Python client (pip install alerce) to query ALeRCE data programmatically. Provides methods for querying objects by class, getting detections, lightcurves, features, classifications, and performing crossmatches. Documentation at alerce.readthedocs.io.

Web Interfaces

ALeRCE Explorer (ztf_explorer) at alerce.online — Browse classified objects, view lightcurves, stamps, classification probabilities, and crossmatch data.

SN Hunter (sn_hunter) at snhunter.alerce.online — Vet supernova candidates ranked by stamp classifier score. Used by the ALeRCE team and collaborators to report discoveries to TNS.

Real-Time Access

Kafka-Connection-Docs — If you want to process ALeRCE-classified alerts in real time, this repo shows how to connect to the output Kafka streams (stamp_classifier and lc_classifier topics). Topics retain data for 48 hours.

Supernova Reporting

TNS_upload — Step-by-step guide for reporting ALeRCE-identified SN candidates to the Transient Name Server.

For ALeRCE Developers & Contributors

If you are contributing to the ALeRCE codebase, the key development repositories are:

Core Development

pipeline — The monorepo containing all pipeline steps, shared libraries (in libs/), Helm charts (in charts/), and CI configuration. Each step is a self-contained Python package with its own Dockerfile. Development follows a trunk-based workflow on the main branch with automated semantic versioning (5000+ releases).

web-services — The API backend. JavaScript/Node.js. Provides REST endpoints consumed by the Explorer, SN Hunter, and Python client.

ztf_explorer and sn_hunter — Vue.js frontends. The Explorer is the main public interface; the SN Hunter is a specialized vetting tool.

Evolution: From Microservices to Monorepo

ALeRCE originally used a microservices architecture where each pipeline step was a separate repository (e.g., feature_step, xmatch_step, prv_candidates_step) built on the APF (Alert Processing Framework). Around 2023, these were consolidated into the pipeline monorepo for easier dependency management, shared library development, and unified CI/CD. The archived standalone step repos remain for historical reference.

Key Libraries (inside monorepo)

lc_classifier — Feature extraction and classification models. P4J — Period finding via information theory. mhps — C-based multiharmonic periodogram. alerce_classifiers — Trained model weights (git submodule). libs/ — Shared utilities for database access, Kafka interaction, and schema definitions.
